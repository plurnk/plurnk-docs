# `run://` — sibling agent runs

Sister agent runs in this session. `run://self` is you; `run://<name>` is a sibling. `WORK(run://<name>):task` SPAWNS a new worker sibling named `<name>`, seeded with that task; `FORK(run://<name>):task` FORKS — branches *you* into a named clone that copies your log so far and continues with the task; `SEND(run://<name>):msg` messages a sibling, waking it if idle; `KILL(run://<name>)` ENDS a sibling. Siblings share this session's files and entries; only the conversation log is private to each. The run ENTITY is created with WORK/FORK, never EDIT — `EDIT(run://<name>)` on the bare run is rejected.

**Spawn a NAMED worker; don't fork yourself for delegation.** "Have a separate worker do X" means spawn a fresh, named sibling — `WORK(run://db-worker):look up db`, `WORK(run://pool-worker):look up pool` — each with its OWN name so you can address it (collect/`KILL`) individually. A worker starts with an EMPTY log (just its task). Forking (`FORK(run://<name>)`) is for branching *your own* reasoning down an alternate path — it clones your entire log so the branch carries your context, not a fresh task. For fan-out, WORK distinct names, one per job.

The slash count is the discriminator: **control** addresses the run by NAME as authority — `run://<name>`, two slashes, no path. **Storage** addresses an entry — `run://<name>/path` reads a sibling's private scratch, `run://self/path` writes your own. So `WORK(run://worker):…` spawns a worker; `EDIT(run://self/todo.md):…` writes a note to your own scratch.

**Delegation workflow — spawn once, hibernate, collect.** To hand work off: `WORK(run://worker):<task>` spawns the worker, then `SEND[202]:<why>:SEND` hibernates you. You wake automatically when the worker concludes — its result arrives **open** in your log as a `SEND` from `run://worker` (a collect-delta); read it and continue. You can also pull on demand: `READ(run://worker)` returns the worker's result, or `425` if it's still running (hibernate and wait). Spawn each worker exactly once: a worker you already spawned keeps running until it wakes you, so re-spawning it — or spawning more while you wait — piles up live runs and trips the active-run cap (`508`). The loop is always WORK → `SEND[202]` → collect on wake, never spawn-again. Fan-out is the same shape: WORK all the workers (distinct names), hibernate once, and each conclusion wakes you with its delta.

**Concluding when you hold live workers.** A `SEND[200]` is refused (`409`) while you still hold a live worker run or open stream — you can't declare done with work outstanding. The system packet lists what you hold live (`## Plurnk Service Child Runs`, `## Plurnk Service Child Streams`); each is addressable. Either `SEND[202]` to hibernate until they conclude, or `KILL(run://<name>)` the ones you no longer need, then terminate. If you already have your answer and the workers are redundant, KILL them and `SEND[200]`.
